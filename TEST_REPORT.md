## 五、性能可优化建议

### 5.1 🔴 高优先级优化

#### 1. 资源体积优化（5.65MB → 建议 < 2MB）

**问题：** 首页总传输大小约 5.65MB，对于移动端用户来说加载压力较大。

**优化建议：**

- **图片资源压缩**
  - 检查项目中的图片资源，使用 WebP 或 AVIF 格式替代 PNG/JPG
  - 对零食图片进行懒加载（lazy loading）
  - 为不同屏幕尺寸提供响应式图片（srcset）

- **代码分割（Code Splitting）**
  - 按路由进行代码分割，首屏只加载首页所需代码
  - Recharts 图表库按需加载（只有详情页和记录页用到）
  - 运动方案、健康页等非核心页面可延迟加载

- **Tree Shaking 优化**
  - 确保 Tailwind CSS 使用 purge 移除未使用样式
  - 检查 lucide-react 图标库是否只导入使用到的图标

#### 2. 资源请求数量优化（54个 → 建议 < 30个）

**问题：** 首页有 54 个资源请求，可能存在较多小文件。

**优化建议：**

- **字体优化**
  - 如果使用了自定义字体，考虑使用系统字体栈或仅加载必要字重
  - 使用 font-display: swap 避免 FOIT（Flash of Invisible Text）

- **第三方依赖合并**
  - 审查是否有可合并的小体积第三方库
  - 考虑将常用工具函数内联

### 5.2 🟡 中优先级优化

#### 3. 首屏渲染优化

**问题：** FCP 指标显示为 0ms，可能存在 SSR/ hydration 问题

**优化建议：**

- 添加骨架屏（Skeleton）或加载占位符，提升感知性能
- 关键路径 CSS 内联（Critical CSS）
- 预加载关键资源（preload）

#### 4. 图片懒加载

**问题：** 首页展示多个零食卡片，可能一次性加载所有图片

**优化建议：**

- 使用 `loading="lazy"` 对首屏外的零食图片进行懒加载
- 使用 Intersection Observer API 实现更精细的懒加载控制
- 考虑使用渐进式图片（blur-up 技术）

### 5.3 🟢 低优先级优化

#### 5. 缓存策略优化

**优化建议：**

- 配置 Service Worker 实现离线访问
- 对静态资源设置合理的缓存策略（Cache-Control）
- 使用 HTTP 缓存头优化重复访问体验

#### 6. 数据预加载

**优化建议：**

- 路由跳转前预加载下一页数据（Link 组件 hover 时预取）
- 热门零食数据可在空闲时间预加载
- 收藏数据使用 IndexedDB 替代 localStorage 提升性能

#### 7. 交互体验优化

**优化建议：**

- 搜索功能添加防抖（debounce），减少不必要的计算
- 份量滑块添加节流（throttle），优化拖动时的渲染性能
- 分类切换时添加过渡动画，使切换更平滑

### 5.4 性能优化效果预估

| 优化项 | 预期提升 | 实现难度 |
|--------|---------|---------|
| 图片压缩 + WebP | 加载速度提升 30-50% | ⭐ 低 |
| 代码分割 | 首屏 JS 减少 40-60% | ⭐⭐ 中 |
| 懒加载 | 首屏请求减少 50%+ | ⭐ 低 |
| 骨架屏 | 感知性能提升明显 | ⭐⭐ 中 |
| 缓存策略 | 二次访问速度提升 80%+ | ⭐⭐ 中 |
